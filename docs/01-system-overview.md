# System Overview

## Purpose

A production-ready web application built for longevity and maintainability. The stack is chosen to minimize operational complexity: one compiled binary, one database, no runtime dependencies beyond the OS. All dynamics are handled server-side with HTMX partial swaps rather than a heavy frontend framework.

## Technology Stack

| Layer | Choice | Rationale |
|---|---|---|
| Language | Go (stdlib-first) | Single compiled binary, excellent concurrency, strong stdlib |
| Frontend dynamics | HTMX 4.0 | Server-rendered partials; no build step, no JS framework |
| CSS | TailwindCSS + DaisyUI | Utility-first; DaisyUI provides themed components without custom CSS |
| Templating | `html/template` (stdlib) | Auto-escaping by default; prevents XSS at the framework level |
| Database | PostgreSQL | Reliable, well-supported; accessed via `pgxpool` |
| DB access | sqlc + pgx/v5 | Type-safe, compiled queries; no ORM magic |
| Migrations | golang-migrate | SQL-first, version-controlled schema changes |
| Auth tokens | JWT (HS256) + opaque refresh tokens | Short-lived access + long-lived rotated refresh |
| UUID generation | google/uuid (UUIDv7) | Time-ordered, collision-resistant IDs |
| Password hashing | bcrypt (cost 12) | Industry standard; cost factor tuned for security/performance |
| Logging | `log/slog` (stdlib) | Structured; JSON in production, text in development |
| Deployment | Docker (multi-stage) | Minimal runtime image; secrets via env vars only |

## Module Boundaries

The codebase is organized around a strict layering rule:

```
HTTP Handlers  в†’  Service Layer (internal/app/)  в†’  Repository (internal/db/)  в†’  PostgreSQL
```

- **Handlers** are thin: parse input, call a service method, render output. No business logic.
- **Services** (`internal/app/`) orchestrate: validate business rules, call one or more repository functions, return domain types.
- **Repositories** (`internal/db/`) are generated by sqlc: pure data access, no logic.
- **Domain types** flow upward; database types stay in `internal/db/`.

Cross-cutting concerns (auth, logging, CSRF, rate limiting) live in middleware and are composed on the router.

## main.go Responsibilities

`cmd/app/main.go` performs only bootstrap and wiring вЂ” no business logic ever touches it:

1. Parse and validate all environment variables into a typed `Config` struct
2. Initialize `pgxpool.Pool` with configured limits
3. Construct the sqlc `Queries` instance
4. Construct service instances, injecting the DB pool
5. Construct the HTTP router, injecting services and middleware
6. Start `http.Server` with the configured timeouts
7. Block on OS signal (`SIGTERM` / `SIGINT`) via `signal.NotifyContext`
8. On signal: log shutdown start, call `server.Shutdown(ctx)` with 30s deadline, stop background workers, close DB pool, log completion

## HTTP Server Timeouts

| Setting | Value |
|---|---|
| ReadTimeout | 10 seconds |
| WriteTimeout | 30 seconds |
| IdleTimeout | 120 seconds |
| ReadHeaderTimeout | 5 seconds |

These values guard against slow-client attacks without penalizing legitimate users on slow connections.
